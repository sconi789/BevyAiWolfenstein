## 项目目的（为什么做）

本项目的目标是**验证 “Bevy + AI 提示词协作” 能否在可控成本下做出一款可玩的 3D FPS 原型**，并形成可复用的方法：用提示词驱动迭代，在短周期内得到“能演示、能回归”的可玩闭环。

## 验证结果（结论）

结论：**可行，并且已达成验证目标**。

在本仓库的实现中，通过提示词驱动迭代，已经完成并跑通了类似《德军总部 3D》风格的基础玩法闭环，包括但不限于：

- **玩法与控制**：行走、转向
- **交互与场景**：界面点击、场景/状态切换
- **渲染与资源**：贴图自适配（UI/分辨率适配思路）、帧动画控制
- **战斗核心**：射线检测（hitscan）、贴花/命中反馈、受击与血量
- **常规系统**：武器、道具、音频实现等

这证明了：在一个纯代码引擎（Bevy）里，AI 提示词可以有效组织“从零到可玩 Demo”的工程化落地。

## 用到的提示词（精炼版，可复用）

以下是本项目中反复验证有效的一组“精炼提示词模板”（你可以按需替换具体数据与资源名）：

### 1) 目标与约束（先锁定范围）

- 你要实现一个“类 Wolf3D 的最小可玩 Demo”
- 必须有：主菜单 → 进入游戏 → 可移动/可射击 → 命中反馈 → 受击扣血 → 返回主菜单
- 不做发散：新增功能必须能演示、能回归，否则拒绝合并
- 资源只用 CC0，并在 `assets/**/README_CC0.txt` 记录来源

### 2) 迭代方式（小步快跑）

- 每次只做一个可验证点（例如：射线命中 + 贴花 + 音效）
- 给出：实现方案、关键代码位置、如何手动验证、失败回滚点
- 保持可运行：每次改动后确保 `cargo run` 能启动并可操作

### 3) 质量门槛（防止“能跑但不可玩”）

- 不能卡死：出生点/出口/障碍生成要避免重叠与堵死
- 输入一致：无输入时速度归零，避免残余速度漂移
- 反馈明确：命中、受击、拾取、开门、传送等要有明显反馈（音效/文字/动画任一即可）

## 对这种模式的期望（下一步愿景）

我们希望把这种“AI 提示词驱动开发”的模式沉淀为一个更稳定、可协作、可持续扩展的流程：

- **更可复用**：沉淀成可复制的提示词与任务拆分模板（从玩法到资产到 QA）
- **更可审计**：每次迭代都能清楚回答“做了什么 / 为什么这么做 / 怎么验证”
- **更可扩展**：在不推翻架构的前提下继续添加：更多武器、更多敌人 AI、更多关卡与系统
